# TestManager — ภาพรวมฟังก์ชันการทำงานของผลิตภัณฑ์

คู่มือฉบับสมบูรณ์เกี่ยวกับสิ่งที่ระบบทำได้ทั้งหมด เขียนขึ้นสำหรับผู้มีส่วนได้ส่วนเสียด้านธุรกิจและผลิตภัณฑ์

---

## TestManager คืออะไร?

TestManager คือแพลตฟอร์มประกันคุณภาพบนเว็บที่มาแทนที่การติดตามด้วยสเปรดชีตแบบแมนนวล ระบบนี้มอบพื้นที่ศูนย์กลางเดียวให้ทีม QA, นักพัฒนา และผู้จัดการในการเขียน test case, รัน test, บันทึก defect, รัน checklist รายวัน, ดำเนินการ user acceptance testing, จัดการ requirements, วางแผนการทดสอบ และจัดการคำขอแบบ ad-hoc — ทั้งหมดจัดระเบียบตาม project

## 1. บัญชีผู้ใช้และการเข้าถึง

ระบบรองรับบัญชีส่วนบุคคลพร้อมการควบคุมการเข้าถึงตาม role ทุกการกระทำจะผูกกับบุคคลที่ดำเนินการ

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **ลงทะเบียนบัญชี** | สร้างบัญชีส่วนตัวด้วยชื่อ, email และรหัสผ่าน |
| **เข้าสู่ระบบ** | ลงชื่อเข้าใช้ด้วย email และรหัสผ่านเพื่อเข้าถึง project ของคุณ |
| **คงสถานะเข้าสู่ระบบ** | Session จะต่ออายุโดยอัตโนมัติในเบื้องหลัง — คุณจะไม่ถูกออกจากระบบโดยไม่คาดคิดระหว่างการทำงาน |
| **ออกจากระบบ** | สิ้นสุด session อย่างปลอดภัยผ่านปุ่มที่ Header bar ด้านบน |
| **แก้ไขโปรไฟล์** | อัพเดตชื่อของคุณได้จากหน้า Profile ซึ่งเข้าถึงได้จาก Header bar ด้านบนขวา |
| **เปลี่ยนรหัสผ่าน** | เปลี่ยนรหัสผ่านของตัวเองได้จากหน้า Profile |

> โปรไฟล์และการออกจากระบบอยู่ที่ **Header bar ด้านบน** ไม่ใช่ใน Sidebar

## 2. Projects

ทุกสิ่งในระบบอยู่ภายใน **Project** project แทนผลิตภัณฑ์, แอปพลิเคชัน หรือขอบเขตการทดสอบ

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **สร้าง project** | ตั้งชื่อ project, รหัสย่อ (เช่น `SHOP`) และคำอธิบายเพิ่มเติม คุณจะกลายเป็น Admin ของ project โดยอัตโนมัติ |
| **ดู project ทั้งหมดของคุณ** | หน้า Projects แสดง project ทั้งหมดที่คุณเป็นเจ้าของหรือได้รับเชิญ พร้อมจำนวนสมาชิกและ test case |
| **เปิด project** | คลิกเข้าไปใน project เพื่อเข้าถึง test case, run, defect, checklist, UAT session, requirements, test plan และรายงาน |
| **แก้ไขรายละเอียด project** | อัปเดตชื่อหรือคำอธิบาย project ได้ตลอดเวลา |
| **ลบ project** | ลบ project และข้อมูลทั้งหมดอย่างถาวร เฉพาะ Admin เท่านั้น |
| **เชิญสมาชิกทีม** | เพิ่มเพื่อนร่วมงานด้วย email และกำหนด role (Admin, Manager, Tester หรือ Viewer) |
| **ลบสมาชิก** | ถอนสิทธิ์การเข้าถึง project ของใครบางคน |

## 3. Test Cases

Test case คือแกนหลักของระบบ — อธิบายว่าต้องทดสอบอะไรและทดสอบอย่างไร

### การจัดระเบียบด้วย Test Suites

Test case ถูกจัดกลุ่มใน **Test Suites** (โฟลเดอร์) Suite สามารถซ้อนกันได้เพื่อสร้างลำดับชั้นที่สะท้อนโครงสร้างผลิตภัณฑ์ (เช่น Login > Forgot Password)

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **สร้าง suite** | เพิ่มโฟลเดอร์ที่มีชื่อเพื่อจัดระเบียบ test case Suite สามารถซ้อนอยู่ใน suite อื่นได้ |
| **เปลี่ยนชื่อหรือย้าย suite** | อัปเดตชื่อของ suite หรือย้ายไปอยู่ใต้ parent อื่น |
| **ลาก-วาง suite** | จัดเรียงลำดับ suite ใหม่ด้วยการ drag-and-drop |
| **ลบ suite** | ลบ suite ที่ว่างเปล่าหรือไม่ต้องการ Test case ที่อยู่ข้างในจะถูกเก็บไว้แต่ไม่ถูกจัดกลุ่ม |

### การเขียน Test Cases

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **สร้าง test case** | เขียน test case พร้อมชื่อ, คำอธิบาย, เงื่อนไขเบื้องต้น, ขั้นตอนการทดสอบ (action + ผลลัพธ์ที่คาดหวัง) และผลลัพธ์ที่คาดหวัง แต่ละ case จะได้รับ ID ที่สร้างอัตโนมัติ (เช่น `DEMO-001`) |
| **กำหนด priority** | ระบุ case ว่าเป็น Critical, High, Medium หรือ Low priority |
| **กำหนด severity** | ระบุความรุนแรงหากเกิดความล้มเหลวในส่วนนี้ |
| **กำหนดประเภทการทดสอบ** | จัดประเภทการทดสอบ (Functional, Regression, Smoke, UAT, Performance, Security เป็นต้น) |
| **กำหนด environment** | ระบุ environment เป้าหมายของการทดสอบ (Staging, Dev, UAT, Production) |
| **ติดตามสถานะ automation** | ระบุว่า case เป็น Manual, Automated หรืออยู่ระหว่างดำเนินการ automation |
| **ติดตามสถานะการ review** | ระบุ case เป็น Draft, Ready, In Review, Approved หรือ Rejected |
| **ติด tag แพลตฟอร์ม** | ระบุว่าการทดสอบครอบคลุม Web Portal, Mobile App, API หรือ Admin Portal |
| **เชื่อมโยงกับ requirement** | บันทึก ID ของ requirement หรือ ticket เพื่อสืบสาว test กลับไปยังความต้องการทางธุรกิจ |
| **มอบหมายนักพัฒนา** | บันทึกว่านักพัฒนาคนใดรับผิดชอบฟีเจอร์ที่กำลังทดสอบ |
| **ระบุความเร่งด่วน** | ระบุ case ว่ามีความเร่งด่วน Normal, High หรือ Critical |
| **เพิ่มหมายเหตุและผลการค้นพบ** | บันทึกข้อเสนอแนะของ QA, การตัดสินใจของ PM และผลการทดสอบบน case นั้น |
| **เพิ่ม Comment** | แสดงความคิดเห็น อภิปราย หรือบันทึกข้อสังเกตบน test case — สมาชิกทุกคนในทีมแสดงความคิดเห็นได้ |
| **แก้ไข test case** | อัปเดตฟิลด์ใดก็ได้บน test case ที่มีอยู่ รวมถึงการแทนที่ขั้นตอนการทดสอบ |
| **ลบ test case** | ลบ test case ที่ไม่ต้องการอีกต่อไป |
| **เรียกดูและกรอง** | กรอง case ตาม suite, สถานะ, priority หรือค้นหาตามชื่อ |
| **Bulk operations** | เลือก test case หลายรายการพร้อมกัน แล้วย้าย Suite, เปลี่ยนสถานะ หรือลบพร้อมกัน |
| **Import test case จาก Excel/CSV** | อัปโหลดสเปรดชีตเพื่อสร้าง test case จำนวนมากแทนการป้อนข้อมูลทีละรายการ *(ดูส่วนที่ 13)* |

## 4. Test Runs และการ Execute

**Test Run** คือ snapshot ของ test case ที่เลือกซึ่งทีมดำเนินการร่วมกัน — ตัวอย่างเช่น regression run ก่อน release

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **สร้าง test run** | ตั้งชื่อ run, คำอธิบายเพิ่มเติม, Sprint และ Version แล้วเลือก test case ที่จะรวมไว้ |
| **Execute test run** | เปิด run และดำเนินการแต่ละ test case ทีละรายการ สำหรับแต่ละ case ให้ระบุเป็น **Pass**, **Fail**, **Blocked** หรือ **Skip** |
| **เพิ่มหมายเหตุการ execute** | เขียนหมายเหตุประกอบผลการทดสอบใดก็ได้ระหว่างการ execute (เช่น อธิบายสิ่งที่ผิดพลาด) |
| **ติดตามความคืบหน้า** | แถบความคืบหน้าแบบเรียลไทม์แสดงจำนวน case ที่เสร็จสิ้นและสัดส่วน pass/fail/blocked |
| **Complete อัตโนมัติ** | เมื่อทุก case ใน run มีผลลัพธ์แล้ว run จะถูกทำเครื่องหมายว่า Completed โดยอัตโนมัติ |
| **ดูประวัติ run** | ดู run ทั้งหมดในอดีต สถานะ Sprint/Version และเวลาที่ complete |
| **ลบ run** | ลบ run ที่ไม่ต้องการอีกต่อไป |

## 5. Defect Log

**Defect Log** คือที่ที่บันทึกและติดตาม bug และปัญหาที่พบระหว่างการทดสอบอย่างเป็นทางการจนถึงการแก้ไข

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **บันทึก defect** | บันทึก bug พร้อมชื่อ, คำอธิบาย, ขั้นตอนการทำซ้ำ, ผลลัพธ์ที่คาดหวังเทียบกับผลจริง, severity, priority และ module แต่ละ defect จะได้รับ ID ที่สร้างอัตโนมัติ (เช่น `DEF-001`) |
| **เชื่อมโยงกับ test case** | เชื่อมโยง defect กับ test case ที่พบปัญหานั้นโดยเป็นตัวเลือก |
| **มอบหมายให้บุคคล** | มอบหมาย defect ให้สมาชิกทีมสำหรับการสืบสวนหรือแก้ไข |
| **ติดตามสถานะ** | เลื่อน defect ผ่านวงจรชีวิต: **Open → In Progress → Fixed → Retest → Verified → Closed** นอกจากนี้ defect ยังสามารถถูกทำเครื่องหมายว่า **Won't Fix** หรือ **Reopened** ได้ตลอดเวลา |
| **ค้นหาและกรอง** | กรอง defect ตามสถานะหรือ severity หรือค้นหาตามชื่อ |
| **แสดงความคิดเห็น** | เพิ่ม Comment เพื่อติดตาม thread การสนทนา และอัพเดตความคืบหน้าการแก้ไข |
| **Export เป็น Excel** | ส่งออกรายการ defect ทั้งหมดเป็นไฟล์ Excel สำหรับรายงาน |
| **ลบ defect** | ลบบันทึก defect ต้องการสิทธิ์ Manager |

**วงจรชีวิต defect โดยย่อ:**

```
Open → In Progress → Fixed → Retest → Verified → Closed
  ↕                    ↑
  Reopened ←───────────┘
  Won't Fix (ได้ทุกขั้นตอน, Manager+)
```

## 6. Daily QA Checklist

**Daily QA Checklist** คือ quality gate แบบเบาบาง — รายการตรวจสอบที่กำหนดไว้ซึ่งทีม QA รันทุกวันก่อน sign-off (เช่น "Smoke test ผ่านแล้ว", "Staging environment เสถียร")

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **ดู checklist วันนี้** | เปิดหน้า checklist เพื่อดูรายการทั้งหมดของวันนี้ที่เตรียมไว้อัตโนมัติทุกวัน |
| **ทำเครื่องหมายรายการว่า Done** | ติ๊กรายการว่าเสร็จสิ้นแล้ว |
| **ทำเครื่องหมายรายการว่า Skipped** | ระบุรายการที่ตั้งใจข้ามไป (เช่น ไม่เกี่ยวข้องในวันนี้) |
| **ทำเครื่องหมายรายการว่า Blocked** | ระบุรายการที่ไม่สามารถเสร็จได้เนื่องจากมีสิ่งกีดขวาง |
| **เพิ่มหมายเหตุให้รายการ** | เขียนหมายเหตุสั้น ๆ อธิบายผลลัพธ์หรือเหตุผลสำหรับรายการใดก็ได้ |
| **ติดตามความคืบหน้าประจำวัน** | แถบความคืบหน้าแสดงจำนวนรายการที่เสร็จสิ้นในวันนั้น |
| **ดูประวัติ session** | ดูบันทึก daily session ในอดีตและจำนวนรายการที่ complete |
| **ปรับแต่ง checklist** | Manager สามารถเพิ่ม, เปลี่ยนชื่อ หรือปิดใช้งานรายการ checklist ให้เหมาะกับความต้องการของ project |

## 7. UAT (User Acceptance Testing)

**UAT session** คือกิจกรรม sign-off อย่างเป็นทางการที่ผู้มีส่วนได้ส่วนเสียทางธุรกิจหรือผู้ใช้ปลายทางตรวจสอบว่าระบบตรงตามความต้องการก่อนที่ release จะได้รับการอนุมัติ

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **สร้าง UAT session** | ตั้งค่า session ที่มีชื่อพร้อมหมายเลขเวอร์ชัน, environment URL, วันที่เริ่มและสิ้นสุด และผู้ติดต่อสนับสนุน แต่ละ session จะได้รับ ID ที่สร้างอัตโนมัติ (เช่น `UAT-001`) ต้องการสิทธิ์ Manager |
| **เพิ่ม test case ใน session** | เลือก test case ที่ควรครอบคลุมใน UAT session นี้ |
| **Execute UAT tests** | ดำเนินการแต่ละ test case และระบุเป็น **Pass**, **Fail** หรือ **Blocked** บันทึกผลจริงและความคิดเห็น |
| **ติดตามความคืบหน้าของ session** | แถบความคืบหน้าแสดงการ complete และสัดส่วน pass/fail/blocked |
| **Sign off session** | Manager สามารถ **Accept** หรือ **Reject** session อย่างเป็นทางการพร้อมหมายเหตุ sign-off session ที่ accepted จะถูกทำเครื่องหมายว่า **Signed Off** ส่วน session ที่ rejected จะถูกทำเครื่องหมายว่า **Rejected** |
| **ดูรายละเอียด sign-off** | ดูว่าใคร sign off, เมื่อไหร่ และหมายเหตุที่ทิ้งไว้ — บันทึกถาวรบน session |
| **Export รายงาน UAT เป็น PDF** | ส่งออกรายงาน UAT session ที่ sign-off แล้วเป็นไฟล์ PDF สำหรับใช้เป็นหลักฐานอย่างเป็นทางการ |
| **ดู UAT session ทั้งหมด** | เรียกดู session ทั้งหมดในอดีตและปัจจุบันพร้อมสถานะและจำนวน case กรองตามสถานะ (Planned, In Progress, Signed Off, Rejected) |
| **ลบ UAT session** | ลบ session ต้องการสิทธิ์ Manager |

## 8. Ad-Hoc Special Cases

**Ad-Hoc Cases** บันทึกคำขอการทดสอบที่ไม่ได้วางแผน, เร่งด่วน หรือเกิดขึ้นครั้งเดียวที่เกิดขึ้นนอกแผนการทดสอบปกติ — ตัวอย่างเช่น คำขอเปลี่ยนแปลงในนาทีสุดท้ายหรือการสืบสวน production แบบ live

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **ส่งคำขอ ad-hoc** | บันทึกปัญหาพร้อมคำอธิบาย, ชื่อผู้ร้องขอ, วันที่ร้องขอ, ระดับความเร่งด่วน (Normal / High / Critical), ระบบต้นทาง, module และการประเมินผลกระทบ แต่ละ case จะได้รับ ID ที่สร้างอัตโนมัติ (เช่น `ADHOC-001`) |
| **ติดตามสถานะ** | เลื่อน case ผ่าน: **Open → In Progress → Resolved** หรือ **Escalated** |
| **บันทึกผลการค้นพบ** | บันทึกแนวทางการทดสอบ, ขั้นตอนที่ดำเนินการ, ข้อมูลการทดสอบที่ใช้, ผลการค้นพบ และการแก้ไข |
| **มอบหมายให้ QA** | มอบหมายการสืบสวนให้สมาชิกทีม QA คนใดคนหนึ่ง |
| **Escalate case** | ระบุ ad-hoc case ว่า Escalated หากต้องการความสนใจจากผู้อาวุโส |
| **แปลงเป็น test case อย่างเป็นทางการ** | หาก ad-hoc issue เผยให้เห็นช่องว่างในการครอบคลุมการทดสอบ ให้แปลงเป็น test case ถาวรด้วยคลิกเดียว บันทึก ad-hoc เดิมจะถูกเก็บไว้และเชื่อมโยงกับ test case ใหม่ |
| **แปลงเป็น defect** | หากพบว่าคำขอ ad-hoc เป็น bug ให้แปลงเป็น defect อย่างเป็นทางการ พร้อมข้อมูลที่กรอกอัตโนมัติจากข้อมูล ad-hoc |
| **ค้นหาและกรอง** | กรองตามสถานะหรือความเร่งด่วน หรือค้นหาตามคำอธิบายและผู้ร้องขอ |
| **ลบ ad-hoc case** | ลบ case ที่ไม่เกี่ยวข้องอีกต่อไป |

## 9. รายงานและการวิเคราะห์

ส่วน **Reports** มอบมุมมองแบบเรียลไทม์เกี่ยวกับสุขภาพด้านคุณภาพให้ผู้จัดการ project และผู้มีส่วนได้ส่วนเสียโดยไม่ต้องค้นหาในบันทึกแต่ละรายการ

| สิ่งที่คุณดูได้ | รายละเอียด |
|-----------------|---------|
| **สถิติสรุป** | จำนวน test case ทั้งหมด, จำนวน test run ทั้งหมด, อัตราผ่านโดยรวม และการแจกแจง case ตามสถานะ (Active, Draft, Archived) |
| **กราฟแนวโน้ม Defect** | กราฟเส้นแสดงจำนวน Defect ที่เปิดเทียบกับที่ปิดรายวัน ย้อนหลัง 30 วัน — ช่วยมองเห็นแนวโน้มคุณภาพในเชิงเวลา |
| **กราฟ Pass Rate Trend** | กราฟแท่งแสดง pass rate ของ 10 Test Run ล่าสุดที่ Completed — แสดงให้เห็นว่าคุณภาพดีขึ้นหรือแย่ลงข้ามแต่ละ sprint |
| **กราฟ Automation Coverage** | กราฟ Pie/Bar แสดงสัดส่วน Test Cases ที่เป็น Manual, Automated และ In Progress automation — ช่วยวางแผนการ automate |
| **กราฟ Requirement Coverage** | กราฟแสดงเปอร์เซ็นต์ Requirements ที่มี Test Cases ผูกอยู่ และเปอร์เซ็นต์ที่ผ่านการทดสอบแล้ว |
| **กราฟประวัติ run** | กราฟแท่งแสดงจำนวน pass, fail, blocked และ skipped สำหรับทุก test run ตามช่วงเวลา |
| **การแจกแจงตาม suite** | ตารางแสดงจำนวน test case ที่แต่ละ suite มี แจกแจงตามสถานะ (Active, Draft, Archived) |
| **พิมพ์/บันทึกรายงาน** | ใช้ฟังก์ชัน Print ของเบราว์เซอร์เพื่อบันทึกรายงานเป็น PDF ได้จากหน้า Reports |

## 10. Requirements Management

**Requirements** คือการบันทึกความต้องการของโครงการและเชื่อมโยงกับ test case เพื่อให้ทีมมองเห็น Traceability — รู้ว่า requirement ใดถูก cover แล้ว และผลการทดสอบเป็นอย่างไร

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **สร้าง requirement** | บันทึก requirement พร้อมชื่อ, คำอธิบาย, Priority และ External ID (เช่น JIRA ticket) แต่ละ requirement จะได้รับ ID อัตโนมัติ (เช่น `REQ-001`) ต้องการสิทธิ์ Manager |
| **แก้ไข requirement** | อัปเดตชื่อ, คำอธิบาย และ Priority ได้ตลอดเวลา |
| **ลบ requirement** | ลบ requirement ที่ไม่ต้องการ ต้องการสิทธิ์ Manager |
| **ผูก Test Case กับ Requirement** | เชื่อมโยง test case หนึ่งหรือหลาย case เข้ากับ requirement เพื่อสร้าง Traceability Matrix |
| **ยกเลิกการผูก Test Case** | ถอด test case ออกจาก requirement เมื่อไม่เกี่ยวข้อง |
| **ดู Coverage Statistics** | ดูสถิติรวม: จำนวน requirement ทั้งหมด, กี่รายการที่มี test case ผูกอยู่, เปอร์เซ็นต์ที่ผ่านการทดสอบแล้ว |
| **ดู Traceability Matrix** | ดูตาราง Requirements x Test Cases ที่แสดงสถานะผลการทดสอบล่าสุดของทุก test case ที่ผูกอยู่ |
| **แนบเอกสาร** | อัปโหลดหรือ link เอกสารประกอบ (เช่น Spec Sheet, Wireframe) เข้ากับ requirement แต่ละรายการ — Tester+ สามารถแนบได้, Manager+ สามารถลบได้ |
| **ค้นหาและกรอง** | ค้นหา requirement ตามชื่อ หรือกรองตาม Priority |

## 11. Test Plans (แผนการทดสอบ)

**Test Plans** คือเอกสารแผนการทดสอบระดับสูงที่กำหนดขอบเขต, ช่วงเวลา และผู้รับผิดชอบสำหรับรอบการทดสอบแต่ละรอบ

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **สร้าง test plan** | ตั้งชื่อ plan, คำอธิบาย, วันที่เริ่มต้น, วันที่สิ้นสุด และสถานะ (Draft, Active, Completed, Archived) ต้องการสิทธิ์ Manager |
| **แก้ไข test plan** | อัปเดตรายละเอียดและเปลี่ยนสถานะ plan ได้ |
| **มอบหมายผู้รับผิดชอบ** | เพิ่มสมาชิกทีมเป็น assignee ของ test plan เพื่อกำหนดความรับผิดชอบ |
| **ลบผู้รับผิดชอบ** | ถอด assignee ออกจาก test plan ได้ |
| **ดูรายการ test plans** | ดู plan ทั้งหมดในโครงการ กรองตามสถานะ |
| **ลบ test plan** | ลบ plan ที่ไม่ต้องการ ต้องการสิทธิ์ Manager |

**สถานะ Test Plan:**

| สถานะ | ความหมาย |
|-------|---------|
| `DRAFT` | กำลังร่างแผน ยังไม่เริ่มดำเนินการ |
| `ACTIVE` | กำลังดำเนินการทดสอบตามแผน |
| `COMPLETED` | แผนการทดสอบเสร็จสิ้นแล้ว |
| `ARCHIVED` | เก็บถาวร ไม่ใช้งานแล้ว |

## 12. Notifications (การแจ้งเตือน)

ระบบ **Notifications** แจ้งเตือนสมาชิกทีมเมื่อมีกิจกรรมที่เกี่ยวข้องกับตน โดยไม่ต้องคอยเปิดดูทุก section เอง

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **ดูการแจ้งเตือน** | คลิกไอคอน Bell ที่ Header bar ด้านบนเพื่อดูรายการแจ้งเตือนทั้งหมด |
| **รับแจ้งเตือน Real-time** | การแจ้งเตือนใหม่จะปรากฏทันทีผ่าน Server-Sent Events (SSE) โดยไม่ต้อง refresh หน้า |
| **ทำเครื่องหมายว่าอ่านแล้ว** | คลิกการแจ้งเตือนเพื่อทำเครื่องหมายว่าอ่านแล้ว หรือทำทั้งหมดพร้อมกัน |
| **ไปยัง entity ที่เกี่ยวข้อง** | คลิกการแจ้งเตือนเพื่อไปยัง defect, test case หรือ review ที่เกี่ยวข้องโดยตรง |
| **ลบการแจ้งเตือน** | ลบการแจ้งเตือนที่ไม่ต้องการออกจากรายการ |

**ประเภทการแจ้งเตือนที่รองรับ:**

| ประเภท | เกิดขึ้นเมื่อ |
|--------|------------|
| `REVIEW_REQUESTED` | มีคนขอให้ review test case |
| `REVIEW_APPROVED` | Test case ที่คุณส่ง review ได้รับการอนุมัติ |
| `REVIEW_REJECTED` | Test case ที่คุณส่ง review ถูกปฏิเสธ |
| `DEFECT_ASSIGNED` | มี defect ถูกมอบหมายให้คุณ |
| `RESULT_ASSIGNED` | มี test result ถูกมอบหมายให้คุณ execute |
| `COMMENT_ADDED` | มีคนแสดงความคิดเห็นใน defect หรือ test case ที่คุณสร้าง |
| `DEFECT_STATUS_CHANGED` | สถานะของ defect ที่คุณเป็นเจ้าของหรือ assignee มีการเปลี่ยนแปลง |
| `TEST_RUN_COMPLETED` | Test run ที่คุณสร้างเสร็จสิ้นแล้ว |

## 13. Import / Upload

ฟีเจอร์ import ช่วยให้ทีมสามารถย้าย test case ที่มีอยู่จากสเปรดชีตเข้าสู่ระบบเป็นจำนวนมาก หลีกเลี่ยงการป้อนข้อมูลด้วยตนเอง

| สิ่งที่คุณทำได้ | รายละเอียด |
|-----------------|---------|
| **อัปโหลดไฟล์** | อัปโหลดไฟล์ Excel (`.xlsx`) หรือ CSV (`.csv`) ที่มีข้อมูล test case |
| **ดูตัวอย่างก่อน import** | ตรวจสอบตัวอย่างข้อมูลก่อนยืนยัน เพื่อให้แน่ใจว่าคอลัมน์ถูกต้อง |
| **Map คอลัมน์** | บอกระบบว่าคอลัมน์ใดในสเปรดชีตของคุณสอดคล้องกับฟิลด์ใด (ชื่อ, ขั้นตอน, priority เป็นต้น) |
| **Import จำนวนมาก** | แถวที่ valid ทั้งหมดจะถูกสร้างเป็น test case ในการกระทำเดียว ระบบจะรายงานจำนวน case ที่ import สำเร็จ |
| **ดาวน์โหลด template** | ดาวน์โหลด Excel template ที่จัดรูปแบบไว้ล่วงหน้าพร้อม header คอลัมน์ที่ถูกต้องเพื่อกรอกและอัปโหลด (ไม่ต้อง Login) |

## 14. สรุป Role และสิทธิ์การใช้งาน

การเข้าถึงภายในแต่ละ project ควบคุมด้วย role ห้าประเภทในระดับระบบ และสี่ประเภทในระดับ project:

### Role ระดับระบบ (Global Role)

| Role | สิ่งที่ทำได้ |
|------|-----------------|
| **Admin** | เข้าถึงได้ทั้งหมดในทุกโครงการ สามารถจัดการผู้ใช้ระดับระบบ เปลี่ยน role และปิดใช้งานบัญชี |
| **Manager** | สร้างและจัดการโปรเจกต์ สร้างและจัดการทุก entity ในโปรเจกต์ที่ตนเป็นสมาชิก |
| **Tester** | เขียน test case, execute test run, บันทึก defect, ส่ง ad-hoc case ในโปรเจกต์ที่ตนเป็นสมาชิก |
| **Developer** | Role สำหรับนักพัฒนา — มีสิทธิ์ Read-only เหมือน Viewer ในระดับโปรเจกต์ ออกแบบมาสำหรับ workflow ของทีมพัฒนาที่ต้องการดู defect และผลการทดสอบที่เกี่ยวข้องกับงานของตน |
| **Viewer** | ดูข้อมูลได้อย่างเดียวในทุก section โดยไม่สามารถสร้างหรือแก้ไขสิ่งใดได้ |

### Role ระดับโปรเจกต์ (Project Role)

| Role | สิ่งที่ทำได้ |
|------|-----------------|
| **Admin** | เข้าถึงได้ทั้งหมดในโปรเจกต์นั้น สามารถจัดการการตั้งค่า project, เชิญ/ลบสมาชิก และลบ project |
| **Manager** | สามารถสร้างและจัดการ test run, UAT session, sign off UAT, จัดการ requirements, สร้าง test plan, ปิดหรือทำเครื่องหมาย Won't Fix สำหรับ defect และปรับแต่ง checklist |
| **Tester** | สามารถเขียนและแก้ไข test case, execute test run, บันทึก defect, อัปเดตรายการ checklist, ส่งคำขอ ad-hoc, ผูก test case กับ requirement และแนบเอกสาร |
| **Viewer** | เข้าถึงแบบอ่านอย่างเดียว สามารถเรียกดูเนื้อหาทั้งหมดแต่ไม่สามารถสร้างหรือเปลี่ยนแปลงสิ่งใดได้ |

> ผู้ใช้สามารถมี role ที่แตกต่างกันในแต่ละ project — ตัวอย่างเช่น Admin ใน project หนึ่งและ Viewer ในอีก project หนึ่ง
>
> ผู้ใช้ที่มี Global Role เป็น DEVELOPER จะมีสิทธิ์เทียบเท่า VIEWER ในระดับโปรเจกต์โดยค่าเริ่มต้น
